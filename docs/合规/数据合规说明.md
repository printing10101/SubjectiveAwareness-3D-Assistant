# 数据合规说明

## 一、数据合规概述

### 1.1 合规目的
本说明旨在明确帮信罪主观明知智能分析系统（以下简称"本系统"）在数据收集、存储、使用、传输全流程中的合规要求和管理措施，确保数据处理活动符合《中华人民共和国数据安全法》《中华人民共和国个人信息保护法》《中华人民共和国网络安全法》等相关法律法规要求。

### 1.2 合规范围
本说明适用于本系统涉及的所有数据处理活动，包括：
- 案件文本数据的收集
- 分析结果数据的生成
- 用户行为数据的记录
- 系统运行数据的存储
- 数据在系统内部的传输
- 数据导出与共享

### 1.3 合规原则
本系统数据处理活动遵循以下原则：
1. **合法性原则**：数据处理活动符合法律法规要求
2. **正当性原则**：数据处理目的正当，不损害用户权益
3. **必要性原则**：仅收集和处理实现目的所必需的数据
4. **最小化原则**：数据处理范围最小化，避免过度收集
5. **透明性原则**：数据处理规则公开透明
6. **安全性原则**：采取必要措施保障数据安全

## 二、数据分类与分级

### 2.1 数据分类

#### 2.1.1 案件数据
- **定义**：用户输入的案件事实文本及分析结果
- **类型**：
  - 原始案件文本
  - 结构化分析结果
  - 推理过程记录
  - 评分与指标数据
- **敏感级别**：高敏感
- **处理要求**：加密存储、访问控制、审计追踪

#### 2.1.2 用户数据
- **定义**：用户账户信息及行为数据
- **类型**：
  - 用户基本信息（用户名、密码哈希）
  - 用户角色与权限
  - 用户操作日志
  - 用户偏好设置
- **敏感级别**：中敏感
- **处理要求**：密码加密存储、操作审计

#### 2.1.3 系统数据
- **定义**：系统运行产生的技术数据
- **类型**：
  - 系统日志
  - 性能指标
  - 错误记录
  - 缓存数据
- **敏感级别**：低敏感
- **处理要求**：定期清理、异常监控

#### 2.1.4 知识库数据
- **定义**：法律知识图谱相关数据
- **类型**：
  - 法律规则
  - 案例数据
  - 标签与分类
  - 关系图谱
- **敏感级别**：中敏感
- **处理要求**：完整性保护、版本管理

### 2.2 数据分级

根据数据敏感程度和重要性，将数据分为四个级别：

| 级别 | 名称 | 描述 | 示例 | 保护要求 |
|------|------|------|------|----------|
| 4级 | 极高敏感 | 泄露会造成严重后果 | 案件原始文本、分析结果 | 加密存储、严格访问控制、完整审计 |
| 3级 | 高敏感 | 泄露会造成较严重后果 | 用户密码、操作日志 | 加密存储、访问控制、审计追踪 |
| 2级 | 中敏感 | 泄露会造成一定影响 | 用户基本信息、知识库数据 | 访问控制、完整性保护 |
| 1级 | 低敏感 | 泄露影响较小 | 系统日志、性能指标 | 基本保护、定期清理 |

## 三、数据收集合规

### 3.1 案件数据收集

#### 3.1.1 收集方式
- **收集渠道**：用户手动输入
- **收集形式**：纯文本格式
- **收集时机**：用户主动发起分析请求时
- **收集范围**：案件事实描述文本

#### 3.1.2 收集限制
- **最小长度**：10字符
- **最大长度**：50,000字符
- **格式要求**：UTF-8编码纯文本
- **内容要求**：案件事实相关文本

#### 3.1.3 收集告知
用户在输入案件文本前，系统显示以下告知信息：
```
本系统为辅助分析工具，您输入的案件文本将用于AI分析。
请注意：
1. 建议脱敏处理个人信息（身份证号、手机号等）
2. 分析结果仅供参考，不构成法律意见
3. 输入数据将加密存储，7天后自动清理缓存
```

#### 3.1.4 收集合法性基础
- **法律依据**：用户明确同意
- **使用目的**：案件分析辅助
- **使用范围**：仅限本系统内部分析
- **存储期限**：分析完成后缓存7天，数据库长期存储（可配置）

### 3.2 用户数据收集

#### 3.2.1 账户信息收集
- **收集内容**：
  - 用户名（必填）
  - 密码（必填，bcrypt哈希存储）
  - 角色（系统分配）
- **收集时机**：用户注册或管理员创建账户时
- **收集目的**：身份认证与权限管理

#### 3.2.2 行为数据收集
- **收集内容**：
  - 登录时间与IP地址
  - 操作类型与时间戳
  - 请求参数（脱敏后）
- **收集方式**：系统自动记录
- **收集目的**：安全审计与异常检测

#### 3.2.3 收集告知
用户注册时，系统显示隐私政策并要求同意：
```
隐私政策摘要：
1. 我们收集您的用户名和密码用于身份认证
2. 我们记录您的操作日志用于安全审计
3. 您的密码使用bcrypt算法加密存储
4. 我们不会向第三方共享您的个人信息
5. 您可以申请删除账户及相关数据
```

### 3.3 系统数据收集

#### 3.3.1 日志数据收集
- **收集内容**：
  - 请求方法、路径、状态码
  - 响应时间
  - 错误信息
  - 用户ID（已登录时）
- **收集方式**：中间件自动记录
- **收集目的**：系统监控与故障排查

#### 3.3.2 性能数据收集
- **收集内容**：
  - CPU、内存使用率
  - API响应时间
  - 缓存命中率
  - 错误率
- **收集方式**：Prometheus监控指标
- **收集目的**：性能优化与容量规划

## 四、数据存储合规

### 4.1 存储架构

#### 4.1.1 数据库存储
- **技术选型**：SQLite（开发）/ PostgreSQL（生产）
- **存储内容**：
  - 用户账户数据
  - 案件数据
  - 分析结果
  - 审计日志
  - 知识库数据
- **存储位置**：本地服务器 / 私有云

#### 4.1.2 文件存储
- **存储内容**：
  - 分析结果缓存（JSON文件）
  - 系统日志（文本文件）
  - 上传文档（临时存储）
- **存储位置**：`.cache/` 目录、`logs/` 目录
- **存储期限**：缓存7天，日志7天

#### 4.1.3 图数据库存储
- **技术选型**：Neo4j（生产）/ 内存图（开发）
- **存储内容**：
  - 法律知识图谱
  - 案例关系图谱
  - 推理路径数据
- **存储位置**：Neo4j数据库文件

### 4.2 存储安全措施

#### 4.2.1 加密存储
1. **密码加密**：
   - 算法：bcrypt
   - 轮数：12轮
   - 盐值：自动生成

2. **敏感数据加密**：
   - 算法：AES-256-GCM
   - 密钥管理：环境变量配置
   - 加密范围：案件文本、分析结果

3. **传输加密**：
   - 协议：HTTPS/TLS 1.3
   - 证书：自签名 / CA签发
   - 加密套件：强加密套件优先

#### 4.2.2 访问控制
1. **数据库访问控制**：
   - 基于角色的权限管理
   - 最小权限原则
   - 数据库连接池管理

2. **文件访问控制**：
   - 文件权限设置（600/700）
   - 目录权限隔离
   - 敏感文件加密

3. **API访问控制**：
   - JWT双Token认证
   - API限流保护
   - IP白名单（可选）

#### 4.2.3 备份与恢复
1. **备份策略**：
   - 全量备份：每日一次
   - 增量备份：每小时一次
   - 备份保留：7天

2. **备份安全**：
   - 备份数据加密
   - 备份异地存储
   - 备份访问控制

3. **恢复测试**：
   - 定期恢复演练
   - 恢复时间目标（RTO）：4小时
   - 恢复点目标（RPO）：1小时

### 4.3 存储期限管理

#### 4.3.1 缓存数据
- **存储期限**：7天
- **清理机制**：自动过期清理
- **清理频率**：每小时检查一次

#### 4.3.2 日志数据
- **存储期限**：7天
- **清理机制**：日志轮转（loguru配置）
- **清理频率**：每日轮转

#### 4.3.3 业务数据
- **存储期限**：长期存储（可配置）
- **归档策略**：历史数据定期归档
- **销毁机制**：按法规要求销毁

## 五、数据使用合规

### 5.1 使用目的限制

#### 5.1.1 案件数据使用
- **使用目的**：
  - 三维度智能分析
  - 相似案例检索
  - 量刑建议生成
- **禁止用途**：
  - 不得用于其他案件分析
  - 不得用于商业目的
  - 不得向第三方共享

#### 5.1.2 用户数据使用
- **使用目的**：
  - 身份认证
  - 权限管理
  - 安全审计
- **禁止用途**：
  - 不得用于用户画像
  - 不得用于精准营销
  - 不得向第三方共享

#### 5.1.3 系统数据使用
- **使用目的**：
  - 系统监控
  - 性能优化
  - 故障排查
- **禁止用途**：
  - 不得用于用户行为分析
  - 不得用于商业目的

### 5.2 使用权限控制

#### 5.2.1 基于角色的访问控制（RBAC）
| 角色 | 权限范围 | 数据访问 |
|------|----------|----------|
| 管理员 | 全部权限 | 所有数据 |
| 检察官 | 分析、查询、导出 | 本人案件数据 |
| 普通用户 | 查询、查看 | 公开数据 |

#### 5.2.2 数据访问审计
- **审计范围**：所有数据访问操作
- **审计内容**：
  - 访问用户
  - 访问时间
  - 访问数据类型
  - 访问操作类型
- **审计日志保留**：7天

### 5.3 数据脱敏

#### 5.3.1 脱敏场景
- 日志记录时脱敏
- 数据导出时脱敏
- 测试数据脱敏

#### 5.3.2 脱敏规则
| 数据类型 | 脱敏规则 | 示例 |
|----------|----------|------|
| 身份证号 | 保留前3后4 | 110***********1234 |
| 手机号 | 保留前3后4 | 138****1234 |
| 银行卡号 | 保留后4位 | ************1234 |
| 姓名 | 保留姓 | 张* |

## 六、数据传输合规

### 6.1 传输安全

#### 6.1.1 传输加密
- **协议要求**：HTTPS/TLS 1.3
- **证书要求**：CA签发证书（生产环境）
- **加密套件**：强加密套件优先
- **前向保密**：启用前向保密

#### 6.1.2 传输完整性
- **完整性校验**：HMAC校验
- **防篡改机制**：数字签名
- **重放防护**：时间戳 + nonce

### 6.2 传输限制

#### 6.2.1 跨域传输
- **CORS配置**：白名单机制
- **允许域名**：
  - 开发环境：`http://localhost:5173`, `http://127.0.0.1:5173`
  - 生产环境：配置实际前端域名
- **禁止通配符**：生产环境禁止使用`*`

#### 6.2.2 数据出境
- **本地化部署**：数据不出本机
- **禁止出境**：不向境外传输数据
- **合规审查**：数据出境安全评估

## 七、数据共享与披露

### 7.1 数据共享原则
- **最小化原则**：仅共享必要数据
- **合法性原则**：共享目的合法
- **安全性原则**：共享过程安全可控

### 7.2 数据共享场景

#### 7.2.1 内部共享
- **共享范围**：系统内部模块间
- **共享方式**：API调用
- **安全措施**：JWT认证、限流保护

#### 7.2.2 外部共享
- **共享原则**：原则上不向第三方共享
- **例外情况**：
  - 法律法规要求
  - 司法机关依法调取
  - 用户明确同意

### 7.3 数据披露

#### 7.3.1 披露原则
- **合法性原则**：依法披露
- **必要性原则**：仅披露必要信息
- **最小化原则**：披露范围最小化

#### 7.3.2 披露场景
- **司法调取**：凭有效法律文书披露
- **安全事件**：依法向监管部门报告
- **用户同意**：经用户明确同意后披露

## 八、用户权利保障

### 8.1 知情权
- **告知内容**：
  - 数据收集目的、方式、范围
  - 数据存储期限、安全措施
  - 数据使用情况、共享情况
  - 用户权利与救济渠道
- **告知方式**：
  - 隐私政策公示
  - 输入前告知提示
  - 注册时同意确认

### 8.2 访问权
- **访问内容**：
  - 个人信息
  - 案件数据
  - 分析结果
- **访问方式**：
  - 系统界面查询
  - API接口查询
  - 导出功能

### 8.3 更正权
- **更正范围**：
  - 个人信息
  - 案件数据（有限更正）
- **更正方式**：
  - 系统界面修改
  - 申请人工更正

### 8.4 删除权
- **删除场景**：
  - 用户注销账户
  - 数据超出存储期限
  - 用户撤回同意
- **删除方式**：
  - 系统自动删除
  - 申请人工删除
- **删除确认**：删除后通知用户

### 8.5 撤回同意权
- **撤回方式**：
  - 系统设置中撤回
  - 书面申请撤回
- **撤回后果**：
  - 停止相关数据处理
  - 可能影响服务使用

### 8.6 救济渠道
- **内部渠道**：
  - 联系客服
  - 投诉举报
- **外部渠道**：
  - 向监管部门投诉
  - 依法提起诉讼

## 九、数据安全事件处置

### 9.1 事件分级

| 级别 | 名称 | 描述 | 响应时间 |
|------|------|------|----------|
| 1级 | 特别重大 | 大量敏感数据泄露 | 1小时内 |
| 2级 | 重大 | 较多敏感数据泄露 | 2小时内 |
| 3级 | 较大 | 少量敏感数据泄露 | 4小时内 |
| 4级 | 一般 | 非敏感数据泄露 | 24小时内 |

### 9.2 处置流程

#### 9.2.1 事件发现
- **发现渠道**：
  - 系统监控告警
  - 安全审计发现
  - 用户举报
  - 外部通报
- **发现后立即**：
  - 记录事件信息
  - 初步评估影响
  - 上报安全负责人

#### 9.2.2 事件响应
- **立即措施**：
  - 隔离受影响系统
  - 阻止数据继续泄露
  - 保护现场证据
- **调查分析**：
  - 确定泄露范围
  - 分析泄露原因
  - 评估影响程度
- **处置措施**：
  - 修复安全漏洞
  - 恢复系统运行
  - 通知受影响用户

#### 9.2.3 事件报告
- **内部报告**：
  - 向管理层报告
  - 向法务部门报告
- **监管报告**：
  - 依法向网信部门报告
  - 依法向公安部门报告
- **用户通知**：
  - 通知受影响用户
  - 提供补救建议

### 9.3 事后处理
- **总结复盘**：
  - 分析事件原因
  - 总结处置经验
  - 提出改进措施
- **整改完善**：
  - 修复安全漏洞
  - 完善安全措施
  - 更新应急预案
- **责任追究**：
  - 追究相关责任
  - 落实整改措施
