# 说明书

# 缘法而智——帮信罪主观明知动态建模与辅助分析系统

---

## 技术领域

本发明涉及人工智能与司法智能化技术领域，具体涉及一种面向**帮助信息网络犯罪活动罪**（以下简称"帮信罪"，对应《中华人民共和国刑法》第二百八十七条之二）中**"主观明知"**核心构成要件的智能审查系统，具体为一种将大语言模型推理、知识图谱检索、可解释评分与司法审查逻辑深度融合的**主观明知动态建模与辅助分析系统**及其方法。

---

## 背景技术

### 一、帮信罪及其"主观明知"要件概述

近年来，帮助信息网络犯罪活动罪（俗称"帮信罪"）已成为我国刑事司法实践中**案件数量上升最快、覆盖面最广**的罪名之一。根据公开统计数据，该罪自 2020 年刑法修正案（十一）正式入罪以来，年均起诉量持续位居各类刑事案件前列。

在刑法理论与司法实践中，"帮信罪"的构成须同时满足四要件：犯罪主体、犯罪客体、主观方面、客观方面。其中，**"主观方面"中的"明知"要件**——即行为人是否"明知他人利用信息网络实施犯罪"——是司法认定中**最关键、最复杂、最易引发争议**的环节。司法解释（《关于办理帮助信息网络犯罪等刑事案件适用法律若干问题的解释》第十一条）虽列举了若干"可推定明知"的情形，但实务中：

1. 行为人常以"不知道""不识字""对方未明示"等理由进行辩解；
2. 案件事实往往高度非结构化（聊天记录、转账流水、电子数据），人工审阅效率低；
3. 不同办案人员对相同证据的"明知"判断存在**主观差异**，案件处理尺度不统一；
4. 资深检察官的办案经验难以系统化沉淀与传递，新人培养周期长。

### 二、现有技术的不足

目前，针对"主观明知"认定的辅助工具主要存在以下不足：

**(1) 关键词检索类工具（如传统类案检索系统）**：
- 仅基于关键词进行字面匹配，难以理解案件事实背后的语义关联；
- 无法对"明知"要素进行**结构化、量化、可解释**的评估；
- 检索结果庞杂，仍需人工二次筛选。

**(2) 单一规则引擎类工具**：
- 规则库静态、僵硬，无法应对新型犯罪手段；
- 缺乏对辩解、矛盾点等**反证**的处理能力；
- 维护成本高，规则冲突时缺乏仲裁机制。

**(3) 通用大语言模型直接调用**：
- 输出"黑箱"，缺乏可解释性与可审计性，难以满足司法严肃性要求；
- 缺乏法律领域知识图谱的约束，易出现**事实幻觉**（hallucination）；
- 未对接案件数据，输出结果与具体案件脱节；
- 部署多为云端调用，**涉密案件数据出网**违反司法数据安全要求。

**(4) 现有司法 AI 系统**：
- 多聚焦于"立案审查""证据校验""量刑计算"等单一环节，**未针对"主观明知"这一具体构成要件**进行专项建模；
- 缺乏"事实抽取 → 多维度评分 → 知识图谱比对 → 辩解合理性评估 → 证据强度溯源"的全链路闭环；
- 缺乏对**多条推理路径冲突**时的冲突检测与可解释仲裁机制。

因此，亟需一种**本地化、可解释、可审计、面向"主观明知"专项**的动态建模与辅助分析系统。

## 发明内容

### 一、要解决的技术问题

本发明的目的在于克服现有技术的不足，提供一种**缘法而智——帮信罪主观明知动态建模与辅助分析系统**（以下简称"本系统"），其解决的技术问题包括：

1. 如何将"主观明知"这一高度抽象的司法判断要素，**结构化、可计算、可追溯**地建模；
2. 如何将大语言模型的语义理解能力与法律知识图谱的领域约束相结合，**避免事实幻觉、提高可解释性**；
3. 如何在**本地化部署**的前提下，完成案件文档解析、多维度评分、类案比对、冲突检测、证据溯源与报告生成全流程；
4. 如何针对"明知"认定中常见的**多路径矛盾、证据强度差异、辩解反证**等复杂情形，提供可仲裁、可审计的推理结果。

### 二、技术方案

#### （一）系统总体架构

本系统采用**五层架构**设计，自上而下分别为：

1. **展示层（Presentation Layer）**：基于 Vue 3 + Vite 的单页应用（SPA），含案件输入、报告展示、知识图谱可视化、实验对比等模块；
2. **接入层（API Gateway Layer）**：基于 FastAPI 的 RESTful API 网关，含 CORS、JWT 鉴权、速率限制、请求日志等中间件；
3. **业务服务层（Service Layer）**：含案件管理、文档处理、分析推理、类案检索、量刑辅助、知识图谱、实验数据等微服务；
4. **AI 推理层（Inference Layer）**：含两阶段推理管线（事实抽取 + 三维度分析）、Prompt 工程系统、分析缓存、Ollama 推理引擎；
5. **数据存储层（Data Layer）**：含关系数据库（SQLite/PostgreSQL）、Neo4j 图数据库、文件缓存、JSON 数据集。

#### （二）"主观明知"动态建模方法

本发明的核心创新在于提出了一种"**主观明知动态建模方法**"，该方法包含以下步骤：

**步骤 S1：案件事实结构化抽取**
- 接收用户输入的案件文本或上传的 PDF/DOCX 卷宗材料；
- 通过文档解析模块提取纯文本内容；
- 调用大语言模型（Ollama + Qwen2.5/DeepSeek-R1 系列）进行**第一阶段事实抽取**，输出结构化 JSON，含：嫌疑人基本信息、客观行为要素、交易细节、沟通记录、辩解内容、证据线索六大字段。

**步骤 S2：三层证据强度评估**
- 系统内置**证据强度评估层（Evidence Strength Layer）**，从直接证据强度、间接证据强度、证据链完整性、证据矛盾度四个子维度，对每一类证据进行 0-10 分量化打分；
- 输出每一证据项的强度标签（强/中/弱/缺位），形成证据强度分布矩阵。

**步骤 S3：三维度"明知"分析模型**
- 依据司法实践与《帮信解释》第十一条，本系统将"主观明知"分解为**三个正交维度**：
  - **维度 D1：客观行为异常度**（Behavior Assessment）：评估嫌疑人行为模式偏离"正常交易"基准的程度，含交易价格异常、频率异常、对接对象异常、行为规避四个子项；
  - **维度 D2：认知能力匹配度**（Cognitive Assessment）：评估嫌疑人个人背景（年龄、职业、学历、相关从业经验）与"应当知道"标准的匹配度，并通过知识图谱检索类案作案模式进行比对；
  - **维度 D3：辩解合理性**（Defense Assessment）：对嫌疑人辩解进行语义一致性、客观证据一致性、内部逻辑一致性三重校验，输出辩解可信度评分与矛盾点列表。
- 三个维度均输出 0-10 分量化评分、推理过程文本与关键依据列表，**支持可解释、可审计**。

**步骤 S4：知识图谱约束与类案匹配**
- 维护**三层知识图谱**：
  - 特征层（Feature Layer）：案件事实特征节点（价格、频率、对象等）；
  - 规则层（Rule Layer）：依据《帮信解释》第十一条构建的"可推定明知"规则节点；
  - 案例层（Case Layer）：真实判决案例节点；
- 节点之间通过"触发""支持""反证""相似"等关系连接；
- 对当前案件进行**Jaccard 相似度 + 语义嵌入**双通道类案检索，返回 Top-K 相似案例与对应"推定规则"。

**步骤 S5：冲突检测与多路径仲裁**
- 当三维度评分出现内部矛盾（例如：行为异常度高，但辩解可信度也高）或知识图谱返回多条相互冲突的推理路径时，**冲突检测器（Conflict Detector）** 自动识别并仲裁：
  - 依据"证据强度优先 → 规则权威性 → 路径可解释性"三级仲裁策略，输出最终推理路径与冲突说明；
  - 全部冲突项、仲裁依据均写入报告，供检察官复核。

**步骤 S6：证据溯源与可解释报告生成**
- 系统对每一个评分、每一条结论，均向上回溯至原始证据片段，生成**证据溯源链**；
- 自动生成结构化分析报告（JSON + HTML + PDF 三种格式），含三维度评分、推理过程、知识图谱推荐路径、相似案例、量刑建议、冲突说明、证据溯源图七大模块。

#### （三）核心算法流程

本发明的核心算法流程可由如下伪代码表示：

```
function Analyze_Subjective_Knowledge(case_text):
    # 步骤 S1：事实抽取
    facts = LLM_Extract_Facts(case_text)
    # 步骤 S2：证据强度评估
    evidence_matrix = Evidence_Strength_Layer(facts)
    # 步骤 S3：三维度评分（并行）
    D1_score, D1_reasoning = Behavior_Assessment(facts, evidence_matrix)
    D2_score, D2_reasoning = Cognitive_Assessment(facts, evidence_matrix, knowledge_graph)
    D3_score, D3_reasoning = Defense_Assessment(facts, evidence_matrix)
    # 步骤 S4：知识图谱类案匹配
    similar_cases = Knowledge_Graph_Retrieve(facts, top_k=5)
    rules_matched = Rule_Matching(facts, knowledge_graph)
    # 步骤 S5：冲突检测与仲裁
    conflicts = Conflict_Detector(D1_score, D2_score, D3_score, similar_cases)
    final_path = Arbitrate(conflicts, evidence_matrix)
    # 步骤 S6：报告生成
    report = Generate_Report(D1, D2, D3, similar_cases, conflicts, final_path, evidence_matrix)
    return report
```

### 三、有益效果

与现有技术相比，本发明具有以下有益效果：

1. **专项建模**：首次将"主观明知"这一抽象法律要件，分解为**三维度（行为异常度、认知匹配度、辩解合理性）**可计算模型，并提供 0-10 分量化评分与推理过程；
2. **可解释性**：每个评分均附带**推理过程文本、关键依据列表、证据溯源链**，非"黑箱"输出，满足司法严肃性要求；
3. **领域知识约束**：通过**三层知识图谱**（特征-规则-案例）对大语言模型输出进行领域约束，**有效降低事实幻觉**；
4. **冲突仲裁**：首次在"明知"分析任务中引入**多路径冲突检测与仲裁机制**，解决了"行为异常 vs 辩解可信"等矛盾情形的可计算化处理；
5. **本地化部署**：通过 **Ollama + 开源大语言模型**实现全流程本地化推理，**案件数据不出网**，满足司法数据安全要求；
6. **全链路闭环**：从文档解析 → 事实抽取 → 证据评估 → 三维评分 → 知识图谱比对 → 冲突仲裁 → 证据溯源 → 报告生成，形成**完整闭环**；
7. **可审计性**：所有用户操作、模型推理、规则匹配、冲突仲裁均产生**结构化日志**，支持事后回溯与责任认定；
8. **可扩展性**：维度、规则、案例、标签均按版本化管理，支持规则库的**热更新与灰度发布**；
9. **可复用性**：三维度模型、证据强度评估、冲突仲裁等核心组件可在其他主观要件（如诈骗罪"非法占有目的"、盗窃罪"秘密窃取"）分析中**复用迁移**。

## 具体实施方式

### 实施例一：系统基础架构部署

**步骤 1.1：环境准备**
- 操作系统：Windows 10+ / Linux / macOS；
- Python 3.10+；Node.js 18+；Ollama 0.3+；
- 内存 ≥ 16GB；推荐独立显卡（GPU ≥ 8GB 显存）。

**步骤 1.2：本地推理引擎部署**
- 安装 Ollama：`ollama serve`；
- 拉取基座模型：`ollama pull qwen2.5:7b` 或 `ollama pull deepseek-r1:7b`；
- 模型通过 HTTP REST API（默认 `http://localhost:11434`）对外提供服务。

**步骤 1.3：后端服务启动**
- 创建虚拟环境：`python -m venv venv`；
- 安装依赖：`pip install -r requirements.txt`；
- 启动服务：`uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload`。

**步骤 1.4：前端服务启动**
- 安装依赖：`npm install`；
- 启动开发服务器：`npm run dev`，默认端口 5173；
- 生产构建：`npm run build`，产物输出至 `frontend/dist/`。

**步骤 1.5：数据库初始化**
- 默认 SQLite，自动建库；
- 迁移：`alembic upgrade head`；
- 种子数据：`python seed_data.py`。

### 实施例二：典型案件分析流程

**场景**：检察官输入一起帮信罪案件文本（含嫌疑人张三、低价收购银行卡 10 张、单张 800 元、有聊天记录、嫌疑人辩称"不知道是用于诈骗"）。

**步骤 2.1：案件文本输入**
- 用户在前端案件输入界面粘贴案件文本（建议长度 200-20000 字符）；
- 或上传 PDF/DOCX 文件，由系统自动解析。

**步骤 2.2：事实抽取**
- 系统调用 LLM 进行第一阶段抽取，输出结构化 JSON：
  ```json
  {
    "suspect": {"name": "张三", "age": 28, "occupation": "无业"},
    "behavior": [{"action": "收购银行卡", "quantity": 10, "unit_price": 800}],
    "communication": [{"type": "微信", "content_summary": "对方明确告知用于电信诈骗"}],
    "defense": "不知道对方用于诈骗",
    "evidence": ["聊天记录截图", "转账记录", "证人证言"]
  }
  ```

**步骤 2.3：证据强度评估**
- 直接证据（聊天记录）：强度 = 9/10（强）；
- 间接证据（价格异常、批量收购）：强度 = 8/10（强）；
- 证据链完整性：9/10；
- 证据矛盾度：2/10（低矛盾）。

**步骤 2.4：三维度评分**
- D1 客观行为异常度：8.5/10（价格显著低于市价 + 批量收购 + 行为规避特征）；
- D2 认知能力匹配度：7.0/10（28 岁成年人 + 微信沟通明确告知 + 无合理不知情理由）；
- D3 辩解合理性：3.0/10（辩解与客观证据严重矛盾）。

**步骤 2.5：知识图谱检索**
- 匹配推定规则：第十一条第（二）项"收到举报后不履行法定职责的"、第（七）项"其他足以认定行为人明知的情形"；
- 推荐相似案例 5 起（Top-5），其中 4 起均认定"明知"。

**步骤 2.6：冲突检测**
- 维度间一致性：D1、D2 高分 vs D3 高分（自相矛盾）→ 检测为"辩解反证型冲突"；
- 仲裁结果：依据"客观证据强度优先"原则，**最终认定：主观明知成立**。

**步骤 2.7：报告生成**
- 输出结构化分析报告（JSON/HTML/PDF）；
- 含三维度评分明细、推理过程、知识图谱推荐路径、相似案例、冲突说明、证据溯源图；
- 检察官可在前端可视化界面查看，并人工复核最终结论。

### 实施例三：边缘案件与多路径冲突

**场景**：嫌疑人李四帮助他人进行"代购"业务，客观上为他人提供了支付结算帮助，但辩称"只是正常代购生意，不清楚上游是诈骗"。

**步骤 3.1**：事实抽取、证据评估同上。

**步骤 3.2：三维度评分**
- D1 客观行为异常度：5.5/10（中等，价格略低、频率正常）；
- D2 认知能力匹配度：4.0/10（中等偏下，李四有正常职业）；
- D3 辩解合理性：6.5/10（辩解与现有证据基本一致）。

**步骤 3.3：知识图谱检索**
- 匹配"代购型帮信"亚类规则，匹配 2 起相似案例；
- 1 起认定"明知"，1 起认定"不明知"。

**步骤 3.4：冲突检测**
- 维度间一致性：D1、D2 中等 + 知识图谱多路径冲突 → 检测为"**多路径仲裁型冲突**"；
- 仲裁策略：调取"代购型"专项规则集，参考辩解与客观证据的匹配度，最终给出**"边缘案例，建议补充侦查"**的建议结论。

**步骤 3.5：报告生成**
- 报告明确标注"边缘案例"、冲突原因、补充侦查建议，**供检察官人工决断**。

### 实施例四：微调模型与版本管理

**步骤 4.1**：使用 Unsloth + LoRA（r=64）对 Qwen2.5-7B 进行监督微调；
**步骤 4.2**：训练数据采用 `instruction-output` 格式，含历史真实判决案例（已脱敏）500-2000 条；
**步骤 4.3**：训练监控采用 TensorBoard，训练完成合并 LoRA 权重并导出 Ollama 模型；
**步骤 4.4**：实验对比模块支持 A/B 测试，新旧模型版本并行运行并对比分析结果；
**步骤 4.5**：所有模型版本、规则版本、案例版本均在知识库生命周期管理模块中版本化管理，支持回滚与灰度发布。

### 实施例五：本地化与司法数据安全

**步骤 5.1**：所有模型、推理、存储均在本地完成，**案件数据不出本机/本局域网**；
**步骤 5.2**：支持离线部署，模型文件预下载后无需联网；
**步骤 5.3**：所有 API 端点强制 JWT 鉴权，操作记录写入审计日志；
**步骤 5.4**：敏感字段（如当事人姓名、身份证号、银行卡号）由脱敏模块在入库前自动脱敏；
**步骤 5.5**：支持国密 SM2/SM4 算法对静态数据加密存储。

### 实施例六：性能与可扩展性

- **响应时间**：单案件分析 ≤ 60 秒（7B 模型 + GPU）；
- **并发能力**：FastAPI 异步架构 + 缓存层，支持 ≥ 10 路并发；
- **案例库规模**：支持 ≥ 10 万案例的知识图谱检索（Neo4j 集群）；
- **可横向扩展**：业务服务层按微服务拆分，可独立部署 Kubernetes 集群。
